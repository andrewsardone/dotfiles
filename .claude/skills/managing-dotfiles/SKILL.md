# Managing Dotfiles

This user tracks dotfiles in a **regular git repository at `~/.dotfiles`**.
Config files are either managed directly by Nix/home-manager modules or
symlinked from the repo into `$HOME` via `dotfile-links.nix`.

## The Setup

- **Repo**: `~/.dotfiles` (regular git repo)
- **Remote**: `https://github.com/andrewsardone/dotfiles.git`
- **Default branch**: `master`
- **Git command**: `git -C ~/.dotfiles` (or just `git` when already in the repo)

Files in `$HOME` are symlinks into `~/.dotfiles` (via `dotfile-links.nix`) or
generated by home-manager modules (zsh, git, tmux, starship). Edit source files
in `~/.dotfiles`, not the symlinks.

## Common Operations

| Task | Command |
|------|---------|
| Check what's changed | `git -C ~/.dotfiles status` |
| Review a diff | `git -C ~/.dotfiles diff` |
| Stage a file | `git -C ~/.dotfiles add <path-relative-to-repo>` |
| Commit | `git -C ~/.dotfiles commit -m "..."` |
| Push (only when asked) | `git -C ~/.dotfiles push` |
| Pull (only when asked) | `git -C ~/.dotfiles pull` |
| Show recent history | `git -C ~/.dotfiles log --oneline` |

## Rules

**Never push automatically.** Push and pull are always explicit — commit and
stop. Only run push or pull if the user specifically asks.

**Edit repo files, not symlinks.** Symlinks in `$HOME` point back into
`~/.dotfiles`; edit the source there so changes are tracked.

## Branching

Most dotfile changes go straight to `master` — tweaks and small iterations
don't need branches. For a larger batch of related changes, a feature branch
makes sense. Follow the user's lead; don't push for PRs on one-liner changes.
